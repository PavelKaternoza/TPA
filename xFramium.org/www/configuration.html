Device Management

The Device Management framework is responsible for acquisition of devices, locking and parallel execution. This library allows access to mobile devices and desktop browsers alike all through a single interface set. Along with device access it also provides artifact production, reporting and integration points for external systems. Using the default Test NG implementation, a develop can quickly gain access to a rich feature set out of the box

XML Configuration

The device registry format is defined using a XML Schema Definition file. This file is located at

http://www.xframium.org/xsd/deviceManagement/deviceRegistry.xsd

Name Description
deviceRegistry: The root element of the device registry
device: Represent a set of device criteria specifying a single device type
  • name: The name of the device used for reporting purposes
  • manufacturer: The device manufacturer - this is used as criteria and for reporting
  • model: The device mode - this is used as criteria and for reporting
  • os: The device os used as criteria
  • osVersion: The device os version used as criteria
  • browserName: The name of the browser used as criteria
  • browserVersion: The version of the browser used as criteria
  • id: A specific device identifier
  • availableDevices: Specifies the number if devices that match this specification. When available devices are greater than one test cases will be split across the devices for faster execution. When less than zero, the system will attempt to read from the cloud provider to find how many devices match
  • cloud: Specifies a cloud name to use overriding the system default cloud
capability: Each device specification can contain a set of named capabilities in addition to those provided above.
  • name: The name of the capability
  • class: Specifies the type of value specified in the text content as STRING,BOOLEAN,PLATFORM or OBJECT)

Data Management

Data-driven testing (DDT) is a term used in the testing of computer software to describe testing done using a table of conditions directly as test inputs and verifiable outputs as well as the process where test environment settings and control are not hard-coded. Data-driven testing is the creation of test scripts to run together with their related data sets in a framework. The framework provides re-usable test logic to reduce maintenance and improve test coverage. Input and result data values can be stored in one or more central data sources. The data comprises variables used for both input values and output verification values. The DDT framework performs "play back" of harvested data producing a powerful automated regression testing tool. While a test may use multiple tables from the data source, only one data source may be used

XML Configuration

When defining page data using XML the data designer will create record type elements. A record type element is the same concept as a database table in that it contains a set of records.

Name Description
recordType: The root element of the application registry
  • name: The name attribute represents the data source name as it will be referenced within the test cases
  • lockRecords: Lock records is a flag (true/false) indicating if a record can be used in parallel. When true a record will be locked by an individual iteration of a test case and released for reuse upon completion otherwise the records can be reused as requested by the test case
record: Represents the application specifications or url for web sites
  • name: The name attribute represents the field name as it will be referenced within the test cases
  • active: Active is a flag (true/false) indicating if this record is active. Setting a record's active flag to false tells the subsystem to ignore it during a run

Excel Configuration

When defining page data using Excel each table is represented by a tab. Each individual field name is specified using the values in the first row of the sheet. You can use a single Excel file for all of the data tables splitting them in using this format. If you want to create a tiered data structures where page data can contain other page data structures then the pipe character will be used. When surrounding a value with the pipe character it indicates that this value is actually a reference to another table. So |authData| indicates that this field is a reference to the authData record type and will contain all of the rows. You can filter the data as well by providing a set of criteria, each contained within a set of square brackets as needed. |authData:[type='user'][age='10']| indicates that this references the authData record type however only the fields that contains the field value of user for type and 10 for age will be selected

Internationalization and localization are means of adapting computer software to different languages however automated tests need to be created to handle the different languages. Content Management is the process of adapting internationalized software for a specific region or language by adding locale-specific components and translating text for use by the automated test scripts. The engine allows for on the fly content replacement for lookups and verification make multi-lingual support easy to implement.

Artifact Management

There are a few points of interest when dealing with Test Artifacts and integrations for reporting and result delivery. There are a handful of prepared reports that are already integrated that can be turned on for automatic delivery. When defining a test run, you can instruct the DataManger instance to automatically download test artifacts at the appropriate times using the setAutomaticDownloads() method. The following artifact types can be used as defined by the ArtifactType enum

Name Description
EXECUTION_REPORT_PDF Gets a copy of the Perfecto Execution report in PDF format
EXECUTION_REPORT_HTML Gets a copy of the Perfecto Execution report in HTML format
EXECUTION_REPORT_CSV Gets a copy of the Perfecto Execution report in CSV format
EXECUTION_REPROT_XML Gets a copy of the Perfecto Execution report in XML format
FAILURE_SOURCE Extracts the state of the device and application. That states are extracted and saved as failureDom.xml to allow for quick debugging of object identification issues
WCAG_REPORT If WCAG operations were used (Branding, contrast, etc.) then this will contain the extracted images and their operational statistics.
DEVICE_LOG Extracts the previous 1000 lines of the device's log file
CONSOLE_LOG Writes the console output for the specified test on the specified device. Only information pertaining to that test/device is stored here
EXECUTION_RECORD_HTML Contains a clean execution report of the test steps that were executed. In addition, a master index file will be created showing the status of all tests executed in this run
EXECUTION_RECORD_CSV Contains an CSV based reports of all test steps that were executed
EXECUTION_TIMING Contains a small timing report of elements marked as timed
